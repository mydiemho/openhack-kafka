Challenge 2 - Azure PaaS Approach

This challenge is a new scenario for a new customer named Fabrikam. Fabrikam has an existing special events ticketing product with an innovative service which offers last minute ticket deals even for the most crowded events.

They are happy with their existing solution but are looking to leverage intelligent cloud services in Azure to enhance the experience. For this reason, the customer is looking to build a proof of concept leveraging Azure platform as a service (PAAS) paradigm.

Scenario

  • The on-prem solution is a microservices architecture where events and telemetry are routed into Kafka. Each time a new ticket is bought, the payment and ticket request message along with application and telemetry data are sent in to the message bus (Kafka).

  • Fabrikam would like to build new services in the cloud to validate payments and ticket availability in near real-time to leverage scaling and expand intelligent cloud aspects of anomaly/fraud detection by performing real-time analytics.

Key requirements from customer

  • Leverage serverless Azure services/PaaS services to keep the infrastructure footprint low.

  • It is important that you integrate the new POC without making any changes to the existing solution which uses Kafka. The existing solution uses Kafka on K8s on-prem.

  • Use of Kappa Architecture to process data as soon as it arrives is preferred to allow quick response to messages.

  • Store the processed results for future visualization and analysis.

  • The customer expects to only pay-as-they-go when possible.

  • Customer is familiar with SQL and would prefer to have a minimum learning curve.

INGESTION

Task 1 - Receiving the Kafka messages on Azure

The real-time message stream needs to be processed in the cloud. It is best practice to have the messages in cloud itself to avoid round trips to on-prem instances. To achieve this you may have to deploy the right Azure service to ingest the messages from on-prem Kafka. Please work as a team to deploy the service that will accept messages from 2 Kafka topics

  1. "payment"
  2. "ticket_request"

Hint: Its important to consider the partitions count to plan for scaling

Coach to verify - Messaging service deployed on the team's Azure subscription

Task 2 - Send on-prem messages to Azure

At this stage you need to plan to get the messages from on-prem to Azure cloud (replicate).

Your team's Azure subscription already has a Kafka cluster running which simulates the on-prem scenario by ingesting different message payloads. Your job, is to send these "on-prem" messages to Azure.

Ask your coach to give you the public IP of the existing Kafka brokers.

Make sure you filter only the desired topics. Currently the management is interested only in the payment and ticket request message processing.

Hint: Kafka Mirror Maker is a popular choice for replicating data but, there are other open source applications that can replicate messages as well. Work as a team to choose the right path.

Task 4 - Enrich the messages on cloud

Continue to build on the previous task's solution, if required, to enrich the messages and use them further in the data pipeline.

You should have a consumer which receives payment messages and another that receives ticket request messages. Simulate a call to a third party application by adding random timeout in your consumers. It is at this stage that raw input messages are enriched with information by adding new data fields. The new fields you can add to enrich data could be

payment message:

{
    "timeProcessed": "2019-05-06 08:10:00.021",
    "paymentValid": "1"
}

ticket_request message:

{
    "timeProcessed": "2019-05-06 08:10:00.021",
    "ticketAvailable": "1"
}

Note: This is a simulation and typically in this stage you would process messages by calling external API's or services and perform checks.

Task 5 - Gaining insights on the data

Discuss with your teammates to complete the rest of the architecture and list out next tasks for the data processing pipeline.

You can refer to the Decision Matrix in the presentation.

Insights to be published:

  2. Show the trend for the valid payments done per minute.

Note - Make sure data is preserved for more than 7 days
